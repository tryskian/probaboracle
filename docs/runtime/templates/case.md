# Case Template

Use this for one representative output, fixed-prompt pulse, eval run, or bounded
slice that shows a Probaboracle seam clearly.

## Metadata

| Field | Value |
| --- | --- |
| Code | `NNN_CASE` |
| Category | `case` |
| Status | `representative`, `anchor`, `retained`, `evicted`, `excluded`, or `archived` |
| Last evidence | `YYYY-MM-DD` |
| Owns | one sentence naming the case this doc stands in for |

## Headline Shape

- `Case: Name`
- or `Representative Case: Name`
- or `Bounded Case: Name`

## Section Order

1. metadata table
2. `Case`
3. `Why It Matters`
4. `Follow-Up`

## Required Case Moves

- name the exact pulse, run, output, or bounded slice
- state the judged object explicitly: the whole pulse/run
- keep the observed shape concrete
- state the verdict plainly
- explain why this case is representative rather than merely memorable

## Default Case Table

| Field | Value |
| --- | --- |
| Artefact | pulse id, run id, output id, or bounded transcript slice |
| Surface | fixed prompt, pulse report, prompt surface, oracle output, drift, or product fit |
| Judged object | one explicit pulse/run lens |
| Observed | compact observed shape |
| Verdict | `pass`, `fail`, `anchor`, `counted_seam`, `excluded_noise`, `retain`, or `evict` |

## Default Why-It-Matters Table

| Signal | Read |
| --- | --- |
| representative shape | one sentence naming the method consequence |

## Case Questions To Answer

- what exact artefact or bounded slice does this stand for?
- what lens judged it?
- what happened in the case?
- why is this one representative of the wider pulse or prompt surface?
- what should happen next because of it?

## Style Rules

- lead with the metadata table
- keep the case table literal and compact
- prefer one representative case over mini-inventories
- keep the interpretation in one small table or one short paragraph
- keep follow-up immediate and operational
